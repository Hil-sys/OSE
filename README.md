# OSE
Курсовая работа на тему "Оптимизация расписания смен сотрудников"

Основная цель этого проекта — разработать алгоритм, который минимизирует отклонение фактического расписания сотрудников от прогнозной сетки при соблюдении некоторых заданных операционных ограничений.

## Описание

### Ключевые сущности

У нас в проекте будут фигурировать три сущности:

| Сущность | Описание | Ключевые атрибуты |
| :--- | :--- | :--- |
| **Сотрудник (Employee)** | Основной ресурс, назначаемый на смены. | `ID`, `Навыки` (например, `chat`, `call`), `Доступность`, `Ограничения по нагрузке`. |
| **Смена (Shift)** | Единица расписания, представляющая собой рабочий интервал. | `ID`, `Временной интервал` (начало и конец), `Приоритет` (опционально). |
| **Прогнозная сетка (Forecast Grid)** | Целевое распределение нагрузки к которой мы стремимся. | - |

### Ограничения и целевая функция

#### **Назначение**

* Ограничения определяют допустимые решения.
* Целевая функция оценивает качество допустимых решений.
* Ошибки в ограничениях → некорректное решение. Ошибки в целевой функции → бесполезное решение.

### Классификация ограничений

#### **Жёсткие (Hard Constraints) — нельзя нарушать:***

1. Доступность — нельзя назначить сотрудника на смену, если он недоступен.
2. Навыки — назначение возможно только при полном соответствии требуемым навыкам.
3. Непересечение смен — один сотрудник не может работать в пересекающихся сменах.
4. Максимальная нагрузка — лимиты часов в день/неделю/месяц не должны превышаться.
5. Укомплектованность смены — смена должна быть укомплектована заданным количеством сотрудников (точное или минимальное).

#### **Мягкие (Soft Constraints) — желательны, но с возможностью штрафа:**

1. Баланс нагрузки — стремление к равномерному распределению нагрузки между сотрудниками.
2. Отклонение от прогноза — минимизация расхождений с прогнозным расписанием.
3. Дополнительные предпочтения — например, минимизация ночных смен, учёт пожеланий сотрудников.

### Целевая функция

Обобщённый вид:
Total Loss = w1 × (Отклонение от прогноза) + w2 × (Дисбаланс нагрузки) + w3 × (Штрафы за мягкие нарушения)

* w1, w2, w3 — настраиваемые веса, позволяющие менять приоритеты (прогноз vs баланс vs комфорт).

### Допустимое vs Оптимальное решение 

* Допустимое — удовлетворяет всем жёстким ограничениям. Уже имеет практическую ценность.
* Оптимальное — допустимое решение с минимальным значением целевой функции. Желательно, но не всегда достижимо быстро.


### Метрики качества и оценка результатов

#### **Назначение метрик**

Позволяют объективно оценить качество расписания:
* Насколько решение хорошее?
* Лучше ли предыдущих версий?
* Какой алгоритм эффективнее?

#### **Классификация метрик (оффлайн, расчёт после построения)**

#### **1. Соответствие прогнозу**

* Отклонение по сменам — разница между фактическим и прогнозным количеством сотрудников в смене.
* Отклонение по человеко-часам — учитывает длительность смен.

#### **2. Баланс нагрузки сотрудников**

* Средняя нагрузка — часов/смен на сотрудника.
* Дисперсия/стандартное отклонение нагрузки — ключевой показатель равномерности распределения.
* Максимальная нагрузка — выявление перегруженных сотрудников.

#### **3. Соблюдение ограничений**

* Количество нарушений soft-ограничений — нежелательные смены, превышение комфортных лимитов.
* Значение целевой функции — суммарный штраф для сравнения алгоритмов.

#### **4. Сравнительные метрики**

Для экспериментов: сравнение алгоритмов (greedy vs ILP), анализ влияния весов в целевой функции.

### Интерпретация и визуализация

* #### **Интерпретация: Метрики анализируются совместно. Хорошее решение — сбалансированное по всем ключевым показателям.**
* #### **Визуализация: Гистограммы нагрузки сотрудников, Heatmap по сменам, Графики загрузки во времени, Сравнительные таблицы алгоритмов**

### Бизнес-ценность, практический импакт и развитие проекта

Проект ориентирован на реальные задачи и имеет практическое применение. Без измеримого импакта даже сложный алгоритм не будет внедрён.

Автоматизация планирования → сокращение времени и ошибок, уменьшение влияния человеческого фактора.

**2. Повышение эффективности использования ресурсов**

Снижение простоев, оптимальная укомплектованность смен, гибкость под изменения нагрузки.

Равномерное распределение нагрузки, контроль перегрузок, прозрачные правила.

**4. Повышение качества обслуживания**

Оптимальная укомплектованность → снижение очередей, повышение скорости обработки, улучшение пользовательского опыта.
